# 3. Service responsibilities

## 3.1 API service

The API service is the public OpenAPI façade.

It owns:

* public user routes
* public account routes
* public transaction routes as façade endpoints
* request validation
* OpenAPI response mapping
* authentication enforcement
* ownership checks
* user profile persistence in PostgreSQL
* account metadata persistence in PostgreSQL
* delegation to Auth service for login path via ALB routing
* delegation to Ledger service for money movement and transaction reads

The API service must not:

* verify passwords directly
* issue JWTs directly
* write auth session records to DynamoDB
* mutate ledger balances
* create ledger transaction rows
* create ledger entries
* own transaction idempotency
* publish Ledger events
* directly write Ledger-owned tables

The API service may:

* validate the public request shape
* verify ownership of user/account resources
* call Ledger service internally
* map Ledger service responses to OpenAPI responses

## 3.2 Auth service

The Auth service owns authentication.

It owns:

* `POST /v1/auth/login`
* password verification
* JWT issuing
* token/session metadata persistence in DynamoDB
* auth readiness checks
* auth-specific logging and redaction

The Auth service reads user credential data from PostgreSQL because users remain in PostgreSQL.

The Auth service stores auth session/token metadata in DynamoDB.

The Auth service must not:

* own user profile data
* own bank accounts
* own ledger accounts
* own transactions
* mutate balances

## 3.3 Ledger service

The Ledger service owns all money movement.

It owns:

* ledger account creation
* ledger account closure
* deposits
* withdrawals
* immutable ledger transactions
* ledger entries
* account balance mutation
* transaction idempotency
* insufficient-funds checks
* maximum-balance checks
* row-level locking/concurrency control
* Ledger outbox event creation
* internal Ledger command API
* internal Ledger query API

The Ledger service is private.

It must not be publicly exposed through the ALB.

The API service calls the Ledger service over private service-to-service communication.

## 3.4 Ledger worker

The Ledger worker is the future asynchronous command processor.

It owns:

* consuming Ledger commands from SQS FIFO when async command processing is enabled
* preserving per-account command ordering
* delegating commands to Ledger domain handlers
* command retry and DLQ behaviour

For this take-home, the synchronous API-to-Ledger path is required for OpenAPI compatibility. The Ledger worker may be implemented as deployable architecture and tested minimally, but the design must be present in Docker/CDK/README.

## 3.5 Ledger event publisher

The Ledger event publisher owns reliable publication of committed Ledger events.

It owns:

* polling Ledger outbox table
* concurrency-safe event claiming
* processing leases
* lease recovery
* event publishing to LocalStack SQS locally
* event publishing to AWS SQS/EventBridge-style target in CDK architecture
* retry with exponential backoff and jitter
* marking events `PUBLISHED`, `FAILED`, or `DEAD`
* structured logs for event publishing

The Ledger event publisher must not:

* process banking commands
* mutate balances
* create ledger transactions
* consume public API requests

# 5. Required OpenAPI corrections

The original OpenAPI contains known defects and omissions. Correct these explicitly.

## 5.1 Authentication endpoint

Add:

```text
POST /v1/auth/login
```

Request body:

```json
{
  "email": "user@example.com",
  "password": "password"
}
```

Response body must include a JWT token according to the chosen auth response schema.

The endpoint is implemented by Auth service.

The OpenAPI file still documents it as part of the single public Eagle Bank API.

## 5.2 Bearer authentication

Add JWT bearer security scheme if missing.

All assessment endpoints are protected except:

```text
POST /v1/users
POST /v1/auth/login
GET /health
GET /ready
```

## 5.3 Password support

Correct `CreateUserRequest`:

* add required `password`
* accept password only on user creation
* never return password
* never return passwordHash

`UserResponse` must never contain password or passwordHash.

## 5.4 Identifier corrections

Correct transaction ID pattern from:

```text
^tan-[A-Za-z0-9]$
```

to:

```text
^tan-[A-Za-z0-9]+$
```

Reason: the original pattern allows exactly one character after `tan-`, while examples use values such as `tan-123abc`.

Convert custom regex definitions incorrectly expressed as `format` into `pattern`.

This applies to:

* `userId`
* `accountNumber`
* `phoneNumber`
* `transactionId`

Preserve standard OpenAPI formats such as:

* `email`
* `date-time`

## 5.5 Create-user 400 response

If `POST /v1/users` has no 400 schema, add:

```yaml
content:
  application/json:
    schema:
      $ref: '#/components/schemas/BadRequestErrorResponse'
```

## 5.6 Transaction amount

Original OpenAPI allows amount `0.00`.

Correct domain rule:

* transaction amount must be greater than `0`
* zero-value deposit or withdrawal returns `400 Bad Request`
* update OpenAPI validation accordingly

## 5.7 Maximum balance

OpenAPI defines maximum account balance as `10000.00`.

Enforce it.

A deposit that would exceed `10000.00` returns:

```text
422 Unprocessable Entity
```

It must not:

* create transaction
* create ledger entry
* mutate balance
* create Ledger event

# 9. DynamoDB auth storage

Users remain in PostgreSQL.

Auth session/token metadata is stored in DynamoDB.

DynamoDB table:

```text
eagle-bank-auth-sessions
```

Local runtime uses DynamoDB Local.

AWS runtime uses DynamoDB.

Recommended table design:

```text
pk = USER#<userId>
sk = SESSION#<sessionId>
```

Attributes:

* `userId`
* `sessionId`
* `tokenId`
* `issuedAt`
* `expiresAt`
* `expiresAtEpoch`
* `revokedAt`
* `lastUsedAt`
* `createdAt`
* `updatedAt`

Enable TTL on:

```text
expiresAtEpoch
```

JWT payload:

* `sub = userId`
* `sid = sessionId`
* `jti = tokenId`
* `iat`
* `exp`

Minimum implementation:

* Auth service issues JWT.
* Auth service stores session metadata in DynamoDB.
* API verifies JWT signature and expiry locally.
* README documents that strict revocation would require API-to-auth-service introspection or short-lived tokens.

Preferred enhanced implementation:

* API verifies JWT signature and expiry.
* API calls Auth service or DynamoDB-backed introspection to verify session state.
* Revoked/missing session returns `401`.

# 14. Account lifecycle and Ledger account creation

When public API creates an account, a corresponding Ledger account must exist before deposits or withdrawals can succeed.

Because the assessment expects create account followed by transaction creation to work predictably, use deterministic synchronous coordination.

Flow:

1. API creates account metadata in PostgreSQL.
2. API calls Ledger service to create Ledger account.
3. Ledger service creates `ledger_account`.
4. API returns OpenAPI account response.

If Ledger account creation fails:

* return a controlled error
* do not leave inconsistent state if practical
* use transaction/compensation where practical
* document the trade-off

Future production evolution:

* API emits `AccountCreated`
* Ledger service consumes it
* account readiness becomes eventually consistent
* saga/process manager coordinates failures

For this assessment, use synchronous coordination and document the future event-driven path.

# 15. Account deletion and immutable transactions

Transactions are immutable financial records.

Physical deletion of an account with existing transactions conflicts with preserving transaction history.

Decision:

* If account has no Ledger transactions, `DELETE /v1/accounts/{accountNumber}` may physically delete metadata and close/remove corresponding Ledger account.
* If account has one or more Ledger transactions, `DELETE /v1/accounts/{accountNumber}` soft-deletes account metadata and closes Ledger account.

Soft-deleted/closed account public behaviour:

* `GET /v1/accounts/{accountNumber}` returns `404`.
* `PATCH /v1/accounts/{accountNumber}` returns `404`.
* repeated `DELETE /v1/accounts/{accountNumber}` returns `404`.
* `POST /v1/accounts/{accountNumber}/transactions` returns `404`.
* `GET /v1/accounts/{accountNumber}/transactions` returns `404`.
* `GET /v1/accounts/{accountNumber}/transactions/{transactionId}` returns `404`.
* `GET /v1/accounts` excludes soft-deleted accounts.

Historical rows remain preserved internally.

A soft-deleted account still counts as an account when evaluating user deletion.

Therefore:

```text
DELETE /v1/users/{userId}
```

returns `409 Conflict` if the user has any account, including a soft-deleted account.

# 17. Synchronous public API façade over Ledger service

The public OpenAPI endpoint expects a normal transaction response, not an async `202 Accepted`.

Therefore implement the public endpoint as a synchronous façade over the private Ledger service.

Flow:

```text
POST /v1/accounts/{accountNumber}/transactions
  -> API authenticates user
  -> API validates request shape
  -> API verifies account ownership
  -> API submits PostTransactionCommand to Ledger service
  -> Ledger service posts transaction atomically
  -> Ledger service creates Ledger outbox event
  -> API maps Ledger result to OpenAPI response
```

Use internal HTTP from API to Ledger service for this take-home.

Production evolution can move command submission to SQS FIFO:

```text
API -> SQS FIFO command queue
Ledger worker -> process command
Ledger worker -> emit result event
API -> wait/poll/correlate result with timeout
```

For the take-home:

* public API remains synchronous
* Ledger service is separated
* Ledger events are event-driven after commit

# 18. Ledger transaction atomicity

Ledger service must use a PostgreSQL transaction.

Deposit flow:

1. Receive command.
2. Validate idempotency.
3. Load Ledger account by `accountNumber`.
4. Lock account row using `SELECT ... FOR UPDATE`.
5. Validate amount > 0.
6. Validate resulting balance <= `10000.00`.
7. Create Ledger transaction.
8. Create Ledger entry.
9. Update Ledger account balance.
10. Store idempotency result.
11. Insert `TransactionPosted` into Ledger outbox.
12. Commit.
13. Return transaction result.

Withdrawal flow:

1. Receive command.
2. Validate idempotency.
3. Load Ledger account by `accountNumber`.
4. Lock account row using `SELECT ... FOR UPDATE`.
5. Validate amount > 0.
6. Validate sufficient funds.
7. Create Ledger transaction.
8. Create Ledger entry.
9. Update Ledger account balance.
10. Store idempotency result.
11. Insert `TransactionPosted` into Ledger outbox.
12. Commit.
13. Return transaction result.

Failure rules:

* insufficient funds returns `422`
* balance limit exceeded returns `422`
* invalid amount returns `400`
* no transaction row is created for rejected commands
* no Ledger entry is created for rejected commands
* no `TransactionPosted` event is created for rejected commands

# 19. Ledger idempotency

Idempotency belongs to Ledger service.

Header:

```text
Idempotency-Key
```

Rules:

* Optional at public API boundary unless OpenAPI requires it.
* Strongly recommended for `POST /v1/accounts/{accountNumber}/transactions`.
* API passes idempotency key to Ledger service.
* Ledger service stores idempotency key with request hash and result.
* Same key and same command returns original result.
* Same key and different command returns `409 Conflict`.

Scope:

* `userId`
* `accountNumber`
* `idempotencyKey`

Store:

* `idempotencyKey`
* `userId`
* `accountNumber`
* `requestHash`
* `status`
* `responsePayload`
* `createdAt`
* `expiresAt`

# 20. Ledger outbox and event publisher

Ledger service writes Ledger events to:

```text
ledger_outbox_events
```

Ledger event publisher publishes those events reliably.

Event statuses:

* `PENDING`
* `PROCESSING`
* `PUBLISHED`
* `FAILED`
* `DEAD`

Ledger event publisher flow:

1. Poll `ledger_outbox_events`.
2. Atomically claim pending/failed events using row locking.
3. Mark claimed events `PROCESSING`.
4. Set processing lease.
5. Publish event to SQS.
6. Mark event `PUBLISHED` on success.
7. Mark event `FAILED` and schedule retry on retryable failure.
8. Mark event `DEAD` after max attempts.
9. Recover expired processing leases.

Claiming must use PostgreSQL semantics equivalent to:

```sql
SELECT ...
FOR UPDATE SKIP LOCKED
```

Delivery semantics:

* at-least-once
* duplicate messages are possible after crashes
* downstream consumers must be idempotent

# 24. Authorization order

User routes:

1. Authenticate.
2. Fetch target user by `userId`.
3. If missing, return `404`.
4. If target user is not authenticated user, return `403`.
5. Continue.

Account routes:

1. Authenticate.
2. Fetch account metadata by `accountNumber`.
3. If missing, return `404`.
4. If soft-deleted, return `404`.
5. If account belongs to another user, return `403`.
6. Continue.

Transaction create/list/detail:

1. Authenticate.
2. Fetch account metadata by `accountNumber`.
3. If missing, return `404`.
4. If soft-deleted, return `404`.
5. If account belongs to another user, return `403`.
6. Call Ledger service.
7. Map Ledger result to OpenAPI response.

* missing required data returns `400`

* own account returns `200`

* soft-deleted account returns `404`

* own account updates
* another user returns `403`
* missing/soft-deleted account returns `404`

* missing account returns `404`

* withdrawal succeeds with funds

* missing account returns `404`
* soft-deleted account returns `404`

* API delegates query to Ledger service
* own account returns transactions
* another user returns `403`
* missing/soft-deleted account returns `404`

`GET /v1/accounts/{accountNumber}/transactions/{transactionId}`:

* API delegates query to Ledger service
* own transaction returns `200`
* wrong accountNumber returns `404`
* another user’s account returns `403`
* missing transaction returns `404`
* missing/soft-deleted account returns `404`

The public API service implements the OpenAPI contract, but it does not own the banking Ledger. Deposits, withdrawals, immutable transaction records, idempotency, and balance mutation are owned by a separate Ledger service. This keeps money movement isolated behind a dedicated boundary.

The Ledger event publisher is a background service that publishes committed Ledger events from the Ledger outbox table to the event queue. It is named after its business responsibility rather than the outbox implementation pattern.
