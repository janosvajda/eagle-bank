Authentication requirement

Implement this chosen authentication endpoint:

POST /v1/auth/login

This endpoint is an implementation choice. The assessment permits one or more authentication endpoints, and this project uses /v1/auth/login as the selected design.

This endpoint must:

* accept credentials using a schema consistent with the user model and OpenAPI conventions
* verify the user credentials
* return a JWT token
* be documented in the OpenAPI specification

Authentication rules:

* POST /v1/users is public.
* POST /v1/auth/login is public.
* Every other endpoint requires a valid bearer token.
* Missing bearer token returns the OpenAPI-defined authentication error response.
* Invalid bearer token returns the OpenAPI-defined authentication error response.
* Password hashes must never be returned in API responses.

Money handling

Inspect the OpenAPI schema before choosing the API representation of money.

The external API must match the OpenAPI specification exactly.

Internally, avoid floating point rounding issues.

Use PostgreSQL Decimal through Prisma Decimal for account balances and transaction amounts unless the OpenAPI specification strongly indicates a different internal representation is more appropriate.

Document the money representation in the README.

Transaction atomicity

Creating a transaction and updating the account balance must happen atomically.

Use a Prisma database transaction.

For deposits:

* create a transaction record
* increase the account balance

For withdrawals:

* verify sufficient funds
* create a transaction record
* decrease the account balance

If the withdrawal has insufficient funds:

* return 422 Unprocessable Entity
* do not create a transaction
* do not update the balance

Endpoint behavior from PDF scenarios

Users

POST /v1/users:

* creates a new user when all required data is supplied
* returns 400 Bad Request when required data is missing

GET /v1/users/{userId}:

* returns the authenticated user’s own details
* returns 403 Forbidden when requesting another user
* returns 404 Not Found when the user does not exist

PATCH /v1/users/{userId}:

* updates the authenticated user’s own details
* returns the updated user details
* returns 403 Forbidden when updating another user
* returns 404 Not Found when the user does not exist

DELETE /v1/users/{userId}:

* deletes the authenticated user when they have no bank accounts
* returns 409 Conflict when the user has at least one bank account
* returns 403 Forbidden when deleting another user
* returns 404 Not Found when the user does not exist

Accounts

POST /v1/accounts:

* creates a bank account for the authenticated user
* returns the created account details
* returns 400 Bad Request when required data is missing

GET /v1/accounts:

* returns only accounts belonging to the authenticated user

GET /v1/accounts/{accountId}:

* returns account details when the account belongs to the authenticated user
* returns 403 Forbidden when the account belongs to another user
* returns 404 Not Found when the account does not exist

PATCH /v1/accounts/{accountId}:

* updates account details when the account belongs to the authenticated user
* returns the updated account details
* returns 403 Forbidden when the account belongs to another user
* returns 404 Not Found when the account does not exist

DELETE /v1/accounts/{accountId}:

* deletes the account when it belongs to the authenticated user
* returns 403 Forbidden when the account belongs to another user
* returns 404 Not Found when the account does not exist

Transactions

POST /v1/accounts/{accountId}/transactions:

* creates a deposit transaction when transaction type is deposit
* increases the account balance for a deposit
* creates a withdrawal transaction when transaction type is withdrawal and funds are sufficient
* decreases the account balance for a withdrawal
* returns 422 Unprocessable Entity when funds are insufficient
* returns 403 Forbidden when the account belongs to another user
* returns 404 Not Found when the account does not exist
* returns 400 Bad Request when required transaction data is missing

GET /v1/accounts/{accountId}/transactions:

* returns transactions for the authenticated user’s own account
* returns 403 Forbidden when the account belongs to another user
* returns 404 Not Found when the account does not exist

GET /v1/accounts/{accountId}/transactions/{transactionId}:

* returns transaction details when the transaction belongs to the specified account and the account belongs to the authenticated user
* returns 403 Forbidden when the account belongs to another user
* returns 404 Not Found when the account does not exist
* returns 404 Not Found when the transaction does not exist
* returns 404 Not Found when the transaction exists but does not belong to the specified accountId

Authorization order

For user routes:

1. Authenticate request.
2. Fetch target user by userId.
3. If user does not exist, return 404.
4. If target user is not the authenticated user, return 403.
5. Continue.

For account routes:

1. Authenticate request.
2. Fetch account by accountId.
3. If account does not exist, return 404.
4. If account exists but belongs to another user, return 403.
5. Continue.

For transaction create and transaction list routes:

1. Authenticate request.
2. Fetch account by accountId.
3. If account does not exist, return 404.
4. If account exists but belongs to another user, return 403.
5. Continue.

For the transaction-detail route only:

1. Authenticate request.
2. Fetch account by accountId.
3. If account does not exist, return 404.
4. If account exists but belongs to another user, return 403.
5. Fetch transaction using both transactionId and accountId.
6. If no matching transaction exists for that accountId, return 404.
7. Continue.
